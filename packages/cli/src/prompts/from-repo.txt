This document will be uploaded to Google Docs as the source of truth for the project. Collaborators will leave inline comments on specific sentences and sections to critique the architecture and drive further work. Optimize the shape of the doc for that use.

## What to cover

Be exhaustive about the big rocks; be selective about implementation detail.

- Every user-facing feature or capability must appear somewhere.
- Every cross-cutting component (something multiple features depend on, or infrastructure not tied to one feature) gets its own section.
- External integrations (APIs, services, auth providers, databases) get called out explicitly.
- Implementation detail is included only when it's load-bearing for understanding how the pieces fit together. Skip line-by-line walkthroughs.

Aim for ~2,000–4,000 words across ≤25 sections. If the repo is too large to cover in that budget, pick the 10–15 most user-visible features and list the rest as bullets under Open Questions — flagging triage beats padding.

Start by reading `README.md`, any `CLAUDE.md`, and the top-level package manifests (`package.json`, `Cargo.toml`, `pyproject.toml`, etc.) — those carry ground truth you shouldn't re-derive from source. Then explore the code.

## Structure

Use this skeleton. Omit sections that genuinely don't apply (e.g., "User journeys" for a pure library), but don't invent alternatives.

1. **Overview** — one paragraph: what this project is and who uses it.
2. **User journeys** — the main end-to-end flows a user or caller goes through. One subsection per journey.
3. **Features** — exhaustive list, grouped into coherent buckets. Lead each bucket with a one-line summary per feature so a skimmer gets completeness before depth; then expand the features that warrant a subsection.
4. **Cross-cutting components** — subsystems that multiple features rely on or that don't belong to any single feature.
5. **External integrations and data** — third-party services, APIs, persistent stores, auth, deployment targets.
6. **Open questions / known gaps** — places where the code is ambiguous, incomplete, disabled-by-default, or where tests and code disagree.

## Faithfulness rules

This doc is ground truth. Commenters will trust it.

- Describe what the code **does**, not what it might have been designed to do. Use present tense.
- Do not invent rationale. If the code doesn't say why, don't speculate about "chosen for performance" or "to ensure consistency."
- If tests and code disagree, describe what the code does and flag the divergence in the Open Questions section.
- If a feature exists behind a disabled flag or is unreachable from any entry point, note it as such rather than presenting it as shipped.
- Describe behavior, not implementation surface. Don't anchor claims to specific class names, function names, or line numbers — those get renamed and refactored, and the doc rots. Refer to a component by its role ("the orchestrator," "the queue store"), not by a concrete identifier (`AgentOrchestrator`, `dequeue()`). File paths are fine when they help a reader find the code (e.g., `packages/foo/src/bar.ts`); class- and method-level pointers are not.

## Shape for commenting

- Keep sections short: 1–3 paragraphs. A commenter can't meaningfully critique a two-page section.
- Prefer atomic claims: one assertion per sentence or paragraph, so someone can comment on exactly the thing they disagree with.
- Write distinctive, quotable sentences. Commenters anchor to text they highlight; avoid boilerplate openings ("This section describes…", "The system handles…") and generic phrases that repeat across sections — they make lousy anchors.
- Use the project's own jargon (the names things have in the code: "the orchestrator," "the runner"), but define each term on first use so a new commenter doesn't have to grep.
- Use consistent H2/H3 hierarchy so the Google Docs outline sidebar becomes a usable table of contents.
- Avoid formatting that is hostile to inline commenting: deeply nested lists (>2 levels), HTML, ASCII diagrams, and narrative code walkthroughs. Short inline code spans and small tables are fine. Short signature-shaped snippets — a type, an interface, a CLI flag list, a config schema — are the most commentable artifact in a spec; permit them when they name something reviewers will want to argue with. Cap each at ~10 lines; skip multi-function code dumps.

## Target

A reader should come away with a complete picture of what the software does in a 15-minute skim. Every feature mentioned; no feature buried in prose. If you find yourself writing something that doesn't help a reader understand the system or doesn't give a commenter something to react to, cut it.
